Problem

Agents can write SQL faster than teams can review the blast radius.

AI agents can now generate and execute database queries as part of normal development workflows. A missing WHERE, an overbroad join, or an accidental catalog read can damage data or expose more than the task required.

Migration tools, ORMs, and database version-control systems do not govern live runtime actions from agents. Teams need a control point before they give agents broader database access.

How it works

One deterministic decision path before Postgres commits.

1

Agent sends SQL

The MCP tool receives SQL plus the agent's stated task.

2

Parse and classify

Statements are parsed with a real Postgres AST, not regex.

3

Policy and simulation

Rules run in memory; risky writes are simulated with timeouts.

4

Allow, block, hold, log

Safe writes can be recorded for undo; risky writes need approval.

Capabilities

SQL classification

Detects reads, writes, DDL, multi-statement batches, catalog access, locking reads, unsafe functions, wrapped writes, and sensitive column exposure.

Policy enforcement

YAML policy blocks unsafe shapes and returns structured reason codes with suggested fixes.

Blast-radius simulation

Risky writes can run inside a timed transaction and rollback to measure exact affected rows before approval.

Approval gates

Large or contradictory writes are held with an approval id and require an operator token before execution.

Audit logs

Every decision is queued to async JSONL audit logs with visibility into queue depth and dropped-record count.

Undo and reversibility

Reversible writes capture before/after images; unsupported shapes are blocked by default rather than committed without undo.

MCP support

The current adapter exposes MCP tools for query execution, approvals, reverts, pending approvals, and audit health.

Latency posture

The pass-through path is benchmarked locally against a 5 ms p99 overhead budget; simulation stays gated to risky writes.

Start in 5 minutes

Run the local developer preview.

This repo is backend-only, so the fastest path is the local Postgres fixture plus the MCP server. The default credentials and policy are for evaluation, not production. 

docker compose up -d
uv sync --frozen
uv run pytest
uv run python -m examples.demo

AGENT_OPERATOR_TOKEN=dev-operator-token \
  uv run python -m adapters.mcp_server

uv run python -m benchmarks.ci_latency_gate

Environment

  • AGENT_DB_DSN targets Postgres.
  • AGENT_POLICY points at YAML policy.
  • AGENT_AUDIT_LOG writes async JSONL logs.
  • AGENT_OPERATOR_TOKEN enables approvals.

Demo scenario

Mass update stopped before commit.

An agent tries UPDATE users SET plan = 'free';. The safety layer detects the missing WHERE, blocks the query, writes an audit record, and returns a machine-readable fix. A scoped bulk update would be simulated first; if it crosses policy thresholds, it is held for operator approval. Reversible writes get an undo_action_id.

What it is not
Not Dolt or database version control.
Not Liquibase, Flyway, Alembic, or migration tooling.
Not an ORM or SQL generation layer.
Not a replacement database or replacement for RLS.

The focus is runtime AI-agent database safety for existing Postgres workflows: preview, policy-check, audit, approval, and undo.

Status

Developer preview.

Built for local evaluation with a seeded Postgres database and MCP adapter. Not yet recommended for production without security review, least-privilege roles, durable/tamper-resistant audit storage, hosted CI, and production deployment hardening.
