Runtime interdiction for Postgres

The safety and undo layer for AI agents touching Postgres.

Interdict classifies agent-generated SQL, enforces policy, measures blast radius, holds risky writes for operator approval, records the audit trail, and preserves an undo path where the write is reversible.

Interdict is not a migration system, not database version control, and not a replacement database. It is a runtime guard for existing Postgres workflows where AI agents can request live database access.

Product specifics

What Interdict actually does at runtime.

Parse

Real Postgres AST

SQL is parsed with the Postgres grammar, so comments, casing, whitespace, aliases, and wrapped writes are handled structurally.

Decide

Deterministic policy

YAML rules produce stable reason codes for unsafe tables, columns, DDL, system catalogs, locking reads, and unbounded writes.

Measure

Blast radius before commit

Risky writes can run inside a time-boxed transaction and roll back, giving the operator an affected-row count before execution.

Recover

Undo when the shape supports it

Reversible writes capture before/after images; unsupported writes are blocked by default instead of pretending they are safe.

Start locally

Run Interdict end-to-end, then connect it to an agent.

Step 1

Install the project environment

Start from the repository root and install exactly from the lockfile. This keeps the parser, MCP server, benchmark tools, and tests on the same versions used by the repo. 

cd /Users/prishar/agent-db-safety
uv sync --frozen
Expected signal

Dependencies install from uv.lock without changing the lockfile.

Requires Docker and uv. The command should finish without editing dependencies.

What the demo proves

The demo creates isolated data, blocks a dangerous mass update, simulates a risky write, records an audit entry, and shows an undo path for a reversible write.

Where to inspect output

Look at terminal responses for blocked, requires_confirmation, simulation, and undo_action_id. Audit records land in logs/audit.jsonl.

How an agent uses it

Register the MCP server, then point the agent at run_query instead of direct database credentials. Human approval uses approve_query with an operator token.

What not to skip

Run the test and latency gates after changing policy behavior. Otherwise a demo can still look fine while the pass-through path, red corpus, or undo assumptions have regressed.

Research signal

Why runtime SQL feedback design matters.

In a controlled study, agents were asked to retry after database writes were denied. The surprising result: when the denial named the exact rule, agents often satisfied the rule literally instead of genuinely reducing the blast radius.

588 closed-loop trials

Three Claude-family models, three database tasks, and four denial styles.

99% first attempts stopped

The tasks reliably triggered Interdict's guardrail before any write committed.

22% -> 95% Haiku evasion jump

Opaque denial to reason-code denial. This is exploratory, not production proof.

scope theater main failure mode

Agents added fake scoping like WHERE TRUE; syntax looked safer, impact did not.

In plain English

If a system only says "you forgot a WHERE clause," an agent may add a WHERE clause that still changes every row. Interdict measures how many rows the query would actually touch, so it can catch the difference between real safety and fake compliance.

Status

Developer preview, not a production deployment recipe.

The problem is production-grade. The current repo is built for local evaluation and research. Production use still needs security review, durable audit storage, persistent approvals, role hardening, and hosted CI.
